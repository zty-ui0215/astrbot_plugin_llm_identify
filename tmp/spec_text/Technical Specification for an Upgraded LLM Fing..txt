Technical Specification for an Upgraded LLM
Fingerprinting and Auditing System for an
AstrBot Plugin
Executive Summary
This specification defines an implementation-ready architecture for a black-box LLM fingerprinting and
auditing system to be implemented by Codex as an AstrBot plugin, with the main user experience
delivered through an AstrBot page panel and a reduced command-only mode used as an auxiliary
fallback. The system is designed for OpenAI-compatible endpoints, but its evidence model is
intentionally broader than protocol conformance because official compatibility layers from Anthropic
and Google can present OpenAI-shaped surfaces while still exposing non-OpenAI behavioral, streaming,
and token-accounting characteristics. The design therefore treats protocol evidence as necessary but
weak, and combines it with knowledge-boundary signals, tokenizer and Unicode behavior, long-form
natural-language behavior, timing and streaming traces, token/context authenticity checks, and
longitudinal stability signals. This direction is consistent with the prior project report’s multi-method
auditing approach and preserves its core idea of cross-checking heterogeneous evidence rather than
relying on a single fingerprint library. fileciteturn(cid:63153)file(cid:63153) 1
The main engineering change in this upgraded design is the replacement of any notion of a static
“reliable fingerprint database” with a Community Fingerprint Corpus plus dynamic probe
generation. Recent work shows that carefully selected active probes can identify model families with
high accuracy, that stable numerical recall near the knowledge boundary is useful for auditing API
substitution and mixed routing, that endpoint identity can drift over time even when uptime and latency
look normal, and that spoofing attacks can make a weaker model imitate a stronger model well enough
to evade overly static fingerprinting pipelines. As a result, the system should store public facts,
discriminative hypotheses, validated probe templates, negative controls, and historical evidence quality
—not just fixed prompts and fixed answers. 2
For implementation, the recommended structure is a host-agnostic audit core plus an AstrBot host
adapter. The audit core owns probe generation, execution, trace capture, feature extraction, evidence
fusion, corpus management, and evaluation. The AstrBot layer owns page-panel rendering, config
binding, task triggering, review workflows, and degraded command entrypoints. This separation keeps
the scientific logic testable outside the host while still making the plugin usable inside AstrBot. The spec
deliberately leaves deployment scale, exact hosting topology, and CI/CD tooling unspecified, as
requested; all such choices are treated as deployment-time concerns rather than assumptions embedded
into the design. 3
Project Scope, Threat Model, and Assumptions
The project goal is not “absolute attestation” of model identity. The goal is to produce a calibrated
audit result that answers five questions: which provider family the endpoint most resembles, which
model family or candidate versions it most resembles, whether the endpoint is likely direct or wrapped
by a proxy/gateway, whether there is evidence of mixed routing or substitution, and whether the

endpoint’s token and context claims are consistent with observed behavior. That framing is aligned
with the direction of recent black-box auditing work, which explicitly treats model identity as a
probabilistic inference problem under adversarial and deployment uncertainty rather than a
deterministic yes/no label. 2
The primary threat model includes honest-but-changing providers, resellers or relays that silently
substitute backends, mixed-routing systems that send only a minority of requests to a different model,
wrappers that add prompts or safety layers, context-summary gateways that make a smaller model look
like a larger-context model, cache-aware intermediaries, and active spoofers that fine-tune a weaker
model to imitate a stronger one. The system must also tolerate protocol-compatible surfaces that
intentionally or unintentionally flatten provider differences. This threat model is motivated by KBF’s
substitution-setting results, Stability Monitor’s findings on identity drift, Anthropic’s and Gemini’s
official OpenAI-compatibility layers, and GhostPrint’s demonstration that spoofing can defeat brittle
fingerprinting procedures. 4
The main assumptions are intentionally narrow. The target endpoint is assumed to accept an OpenAI-
compatible request surface for the plugin’s primary execution path. The design may optionally include
native adapters for reference baselines, but the plugin’s production target remains OpenAI-compatible
APIs. Deployment scale is unspecified. Exact hosting is unspecified. CI/CD is unspecified. The system
therefore avoids assumptions that would require a fixed cluster size, a fixed scheduler, or a fixed release
workflow. It should run on a single process for development and local testing, while also allowing
separate worker and storage processes in larger deployments, but those packaging decisions remain out
of scope for the scientific core.
The system should assume imperfect observability. Some endpoints will not expose complete usage
fields, stable request IDs, or predictable streaming traces. Some will normalize or ignore unsupported
fields. Anthropic’s OpenAI SDK compatibility documentation explicitly states that some OpenAI fields
are ignored, some are hoisted or concatenated, prompt caching is not supported in the compatibility
layer, and fields such as system_fingerprint remain empty there. Gemini’s OpenAI compatibility page
similarly shows that Gemini can be reached through OpenAI libraries with a modified base_url , and
Google recommends the native Gemini API for access to the latest features. OpenAI’s own API overview
emphasizes that response headers, request IDs, limits, and streaming events are part of the observable
surface. These facts justify explicit handling of missingness and compatibility-layer transformations
rather than treating them as random noise. 5

The Host Adapter is AstrBot-specific and should expose a page-panel-first workflow. The page panel is
the primary interface for target registration, suite selection, live execution, evidence review, corpus
browsing, contribution review, and report export. A reduced command mode is retained for operational
convenience and should only expose a minimal set of functions such as “run audit,” “show status,”
“open latest report,” and “submit candidate evidence.” This matches the user requirement that
command-triggered operation be auxiliary while the main experience lives in the page panel.
The Audit Orchestrator must be host-independent. It owns suite resolution, target normalization, audit
reproducibility metadata, probe-family quotas, rate-limit-aware scheduling, retries with jitter, and
separation of “probe generation” from “probe execution.” That separation matters because official
APIs expose rate limits and response headers differently, and because some evidence branches depend
on repeated or delayed sampling. OpenAI documents both rate-limit headers and request IDs; Anthropic
documents separate token-counting limits and Messages API streaming; Gemini’s Interactions API
exposes an SSE-based event model with final usage statistics. The orchestrator should therefore capture
response metadata, timing, and usage as first-class data rather than incidental logs. 6
The Execution & Capture layer should normalize every call into a TraceEnvelope with request payload,
canonicalized response body, streaming event sequence if present, headers, timing metrics, and
extracted server-reported usage. OpenAI recommends logging x-request-id and exposes rate-limit
headers; Anthropic’s streaming documentation describes named SSE event types and its error
documentation notes that oversized requests can be rejected at the Cloudflare layer with 413
request_too_large ; Gemini’s Interactions API streams step-based events and returns final usage in the
interaction.completed event. Those provider-specific behaviors are exactly the sort of differentiating
surface that the system should capture before any normalization throws them away. 7

Community Fingerprint Corpus
The Community Fingerprint Corpus is the project’s long-lived knowledge asset. It must not be modeled
as a flat list of “known prompts.” Instead, it should store public facts, boundary claims, provider quirks,
validated templates, known transformations introduced by compatibility layers, quality statistics for each
evidence artifact, and negative examples. That design directly responds to the empirical weaknesses of
static fingerprints observed in substitution, drift, wrapper, and spoofing settings. 8

Contribution and validation workflow
Every community contribution should go through automated and human review. Automated review
should check schema validity, semantic deduplication, safety class, obvious leakage or self-referential
prompts, answer length bounds, anti-cache parameterization, and initial separation statistics against a
baseline pack. Human review should focus on interpretability, public verifiability of claims, boundary
quality, and whether the contribution creates a perverse incentive toward overfitting a particular
provider snapshot. This workflow is especially important because Stability Monitor shows real endpoint

behavior changes over time and GhostPrint shows that an attacker can optimize against a known
fingerprinting regime. 10
Suggested artifact retirement rules should be explicit. Any probe family whose separation score decays
below threshold for two consecutive corpus releases, whose false-positive rate rises above a configured
bound, or whose performance becomes highly provider-region-sensitive should be downweighted or
retired. Corpus versioning must preserve audit reproducibility: every audit result should record the exact
corpus version, probe-set version, baseline version, and sampler seed used to generate it.
Evidence Collection and Dynamic Probe Design
The system should prioritize evidence sources that are both discriminative and hard to spoof jointly. A
single strong probe may still be informative, but the system should prefer independent evidence
bundles that fail differently under wrappers, routing, caching, and imitation attacks. LLMmap supports
the value of active, carefully chosen probing; KBF supports boundary-oriented questions; Stability
Monitor supports repeated sampling over time; Invisible Traces supports combining static and dynamic
signals; LLMs Have Rhythm and the inference-system fingerprinting work show that timing and
infrastructure leak usable information; and GhostPrint demonstrates why no single branch should
dominate the final decision. 11

Random sampling strategy
To avoid static probing, the system must never replay the same narrow set of prompts on every run. Use a
stratified weighted reservoir with three anti-static penalties:
· a recency penalty against recently used templates;
· a target exposure penalty against prompts already seen by the same target or target cluster;
· a cache-risk penalty against probes whose prefixes are too stable across audits.
For candidate probe i, define selection score
u = log(quality +ϵ)+λ log(separation +ϵ)+λ log(stability +ϵ)−λ recent_use −λ target_exposure −λ cache_risk
i i 1 i 2 i 3 i 4 i 5 i
and sample within each family proportionally to exp(u i ). This keeps strong probes in circulation while
continuously rotating their concrete realizations. Prompt-caching research shows timing can leak prefix
reuse and provider behavior, and OpenAI’s prompt caching documentation confirms a cached_tokens
signal and organization-scoped cache behavior. Those facts make anti-static sampling operationally
important rather than merely aesthetic. 13

Detection methods by evidence branch
Token and protocol detection. OpenAI provides an input-token count endpoint at /v1/responses/
input_tokens , rate-limit headers, x-request-id , and cached_tokens in prompt-token details. Anthropic
provides messages.count_tokens , states that token counting is free and separately rate-limited, and
notes that the count is an estimate. Gemini’s Interactions API returns final usage in the completed
event, and model metadata exposes inputTokenLimit and outputTokenLimit . These observables
support three checks: preflight count versus postflight usage consistency, monotonic slope consistency
across controlled input lengths, and cache-consistency checks under repeated-prefix versus nonce-
mutated runs. 15
Protocol legality and compatibility-layer detection. Anthropic’s OpenAI compatibility layer
documents hoisting of system/developer prompts, unsupported prompt caching, ignored fields, and
empty system_fingerprint ; Gemini documents OpenAI-library access via a Gemini base URL; OpenAI
emphasizes headers, request IDs, and event surfaces. Therefore the adapter should test field support,
unsupported-field handling, message-hoisting signatures, error shape, header presence, and stream
event topology. A protocol branch should classify not only “looks OpenAI-compatible,” but also
“looks like OpenAI-compatible Anthropic,” “looks like OpenAI-compatible Gemini,” or “looks like a
thinner proxy over another implementation.” 16
Behavioral and refusal detection. Retain the earlier project’s behavioral and refusal-style branches,
but run them on both short probes and long-form tasks. The classifier should deliberately separate
“useful behavior” from “identity behavior” so that a wrapper that changes refusal wording but not
deep structure does not dominate the verdict. Long-form structured compliance, section topology,
uncertainty handling, and rewrite style should be weighted more than superficial disclaimer text,
because wrapper prompts can easily alter the latter. This is a direct response to wrapper and spoofing
risks noted in hybrid fingerprinting and spoofing studies. 17
Knowledge-boundary detection. The corpus should store public facts and release-ordering claims that
are stable and verifiable. The detector should ask questions close to the known knowledge boundary,
score correctness where public anchors exist, and separately score uncertainty behavior. KBF’s key
insight is that near-boundary numerical recall and inconsistency behavior can distinguish same-surface
endpoints and detect even minority substitution traffic. The spec should therefore prefer probes that
measure “what the endpoint knows, what it does not know, and how consistently it admits
uncertainty,” rather than only raw factual recall. 18
Tokenizer and Unicode detection. Keep the original token/Unicode family from the previous report and
expand it. Include zero-width characters, confusable characters, mixed-width punctuation, mixed-script
prompts, long escaped JSON/XML segments, repeated-token boundaries, emoji sequences, and exact
roundtrip requests. These cases often survive wrapper prompts because they are rooted in
preprocessing, tokenizer, escaping, and normalization behavior. The inference-system fingerprinting
paper further supports the view that subtle implementation differences can propagate into observable
outputs. 19

Mixture and route-substitution detection
Mixture detection should not be treated as ordinary single-label classification. It should be a latent-
mixture inference problem over repeated responses to the same semantic task under controlled
sampling conditions.
For each repeated probe instance, construct a joint feature vector:
z = [e ,b ,u ,t ,r ,k ]
i i i i i i i
where e i is semantic embedding, b i behavioral structure vector, u i Unicode/tokenizer features, t i timing/
stream features, r i response-format features, and k i knowledge-boundary outcome features.
Then:
1. cluster per-probe repeats using HDBSCAN or a Gaussian mixture;
2. compare cluster centroids to baseline centroids;
3. quantify split stability across time and temperature;
4. compute a mixture score using separation, alternative-baseline affinity, and repeat persistence.
A practical mixture score is:
R = σ(β +β ΔBIC+β Sep+β AltAffinity+β TemporalPersistence)
mix 0 1 2 3 4
where ΔBIC compares K = 1 and K > 1 cluster models, Sep is inter-cluster separation,
AltAffinity measures whether different clusters align to different baselines, and TemporalPersistence
measures whether the split reappears in later runs. KBF’s substitution results strongly support making
this branch explicit because it can detect mixed routing even at low substitution rates when the
substituted traffic is sufficiently separated. 18
Token and context authenticity scoring
Token authenticity should combine:
· count-vs-usage consistency where supported;
· length-gradient slope consistency over (cid:63155)(cid:63158)(cid:63159)/(cid:63158)(cid:63154)(cid:63155)/(cid:63154)(cid:63153)(cid:63155)(cid:63157)/(cid:63155)(cid:63153)(cid:63157)(cid:63161)/... token steps;
· cache consistency under repeated-prefix and prefix-mutation conditions;
· behavioral upper bound from what the model demonstrably used.
OpenAI’s token-count endpoint returns response.input_tokens , OpenAI prompt caching exposes
cached_tokens , Anthropic provides messages.count_tokens but explicitly notes it is an estimate, and
Gemini streaming returns final usage with total and cached token fields in the Interactions API example.
These support a graded authenticity score rather than a brittle pass/fail rule. 20
Context authenticity should separately score hard limit, effective recall depth, and silent truncation
risk. Anthropic’s context-window documentation explicitly states that context includes the full request
plus generated output and warns that more context is not automatically better because of “context
rot”; it also documents distinct behaviors when the context window is exceeded. Gemini model
metadata exposes input and output token limits. Therefore a truthful platform can still show degraded
long-context recall without lying, which means the system must not equate weak recall with fraud. The

Evaluation, Privacy, Defaults, and References
The evaluation harness should be built around reference classes, not just benchmark prompts. At
minimum it should include: direct official endpoints, official compatibility layers, open-source self-
hosted endpoints, transparent pass-through relays, wrapper gateways, summarization gateways, mixed-
routing relays with known proportions, and spoofed or imitation endpoints when feasible. This mirrors
the scientific threat landscape: LLMmap validates active fingerprinting against many versions, KBF
validates substitution and mixed routing, Stability Monitor validates longitudinal identity changes, and
GhostPrint validates adversarial spoofing. 2
A practical test harness should report:
· classification metrics: top-(cid:63154) accuracy, macro-F(cid:63154), AUROC by provider family and model family;
· auditing metrics: mixture detection precision/recall, token-authenticity false-positive and false-
negative rates, context-authenticity false-positive and false-negative rates;
· calibration metrics: ECE, Brier score, reliability curves;
· operational metrics: average probes per audit, average latency per branch, cache-hit rate, and
drift-detection delay.
False positives and false negatives should be analyzed by failure mode, not only by label. Useful buckets
are: compatibility-layer confusion, wrapper-prompt distortion, knowledge-boundary staleness, cache
contamination, stream metadata loss, and spoofing success. Stability Monitor’s drift framing is
especially useful here because some apparent false positives are actually evidence that the endpoint
changed between baseline capture and evaluation. 22
