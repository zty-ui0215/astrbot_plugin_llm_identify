Reliable Detection and Measurement of LLM
Token Counts
Executive Summary
Assumption. No single target LLM, provider, or deployment platform is fixed. This report therefore treats
token counting as a layered systems problem across three cases: open tokenizer artifacts, closed APIs
with provider-native counters, and opaque or partially observable services.
The central conclusion is that there is no platform-independent “true token count” for a request
unless the exact model family, model version, request shape, and tokenizer/formatter path are
known. Official provider documentation now states this explicitly in different ways: OpenAI’s input-
token counter counts the exact payload the model will receive, including formatting tokens for message
roles and boundaries; Amazon Bedrock says token counting is model-specific and that the returned
count matches what would be charged for the same request; Anthropic exposes token counting for
Claude messages; and Google documents both token-count APIs and cases where multimodal counts are
only estimates before execution. 1
For production-grade reliability, the best detector stack is hierarchical. First preference is a provider-
native exact counter for the full payload, or a local tokenizer officially aligned to the target model
when available. Second preference is a static tokenizer reconstruction from released tokenizer assets.
Third preference is a hybrid estimator combining local surrogate tokenizers, dynamic API probing, and
calibration against a small exact-count sample. Uncalibrated heuristics such as “characters divided by
four” are useful only as coarse budget guards, not as billing- or truncation-grade measurements. 2
Recent literature makes clear why this hierarchy is necessary. Modern tokenization remains dominated
by BPE variants, byte-level BPE, WordPiece, and SentencePiece-style BPE/Unigram pipelines, but
recent work also shows that tokenization is more behaviorally consequential than many practitioners
assume. BPE merge lists can reveal training-mixture information; non-canonical tokenizations can
preserve semantics and sometimes improve performance; adversarial tokenizations can bypass safety
systems; and tokenization multiplicity can even create pricing variation for the same output string. 3
For very long texts, the report recommends two distinct strategies. If exact tokenizer assets exist, use
streaming exact counting with seam correction across chunk boundaries. If they do not, use stratified
window sampling plus a control-variate surrogate tokenizer, and report explicit confidence intervals.
Long-context benchmarks such as RULER, ∞Bench, LongBench v2, and Needle-in-a-Haystack should be
used not only to test model reasoning, but also to stress-test the counting system against long inputs,
deep boundary effects, and mixed-format documents. 4
A final practical point is that token counting drift is now operationally real. Anthropic documents that
Claude Sonnet 5 uses a new tokenizer that yields roughly 30% more tokens than Sonnet 4.6 for the same
text, and Google documents that multimodal preflight token counts may differ from final billed usage.
Any counting plugin should therefore be version-aware, continuously calibrated, and able to surface
confidence and disagreement—not just a single integer. 5

Tokenization Algorithms and Sources of Ambiguity
Token counting is not just the output of a tokenizer model. In modern stacks it is the result of a pipeline:
normalization, pre-tokenization, model segmentation, post-processing, and decoding conventions.
Hugging Face’s tokenizer pipeline documentation makes this decomposition explicit; SentencePiece
further embeds normalization rules directly into the model artifact; and GPT-style formats may inject
special structural tokens for messages, channels, boundaries, and tool calls. A counting system that
ignores any one of these layers will often be systematically wrong. 6

Two foundational observations matter for measurement. First, BPE-style counting is attractive because it
is reversible, lossless, arbitrary-text capable, and compressive; OpenAI’s tiktoken notes that in
practice one token corresponds to roughly four bytes on average. Second, SentencePiece’s defaults are
intentionally not transparent to naive text-length reasoning: it uses a custom NFKC variant, can add a
dummy prefix, collapses extra whitespace, and replaces normal spaces with the ▁ meta-symbol. These
design choices are excellent for training and inference consistency, but they create failure modes for
external counters that only inspect raw characters. 13
Where ambiguity enters
The most important ambiguities in token counting are not “algorithm family” questions but pipeline-
state questions.
A first ambiguity is normalization. SentencePiece normalizes with a custom NFKC variant by default and
allows custom rules embedded into the model file itself; these rules are applied before tokenization both
at training and inference time. That means visually similar strings, full-width forms, combining-mark
3

--- PAGE 4 ---
variants, and repeated spaces may collapse into identical normalized inputs even before segmentation is
attempted. 14
A second ambiguity is pre-tokenization. Hugging Face documents that pre-tokenizers may split on
whitespace, punctuation, or even individual digits, and these upstream boundaries constrain
downstream subword choices. A digit-aware pre-tokenizer, for example, can make "911" behave
radically differently from a tokenizer that sees the whole string as one unit. This becomes important for
numerals, identifiers, code, and JSON keys. 15
A third ambiguity is post-processing and structural wrappers. OpenAI’s current token-count guide
says the exact count includes request-structure formatting tokens such as roles and boundaries that do
not appear in the raw user text. In the documented Harmony format for GPT-OSS-style models, structural
tokens such as <|start|> , <|message|> , <|end|> , and <|channel|> are explicitly part of the token
stream. Tool schemas, system messages, and conversation wrappers therefore change counts even if the
natural-language text is unchanged. 16
A fourth ambiguity is version drift. Anthropic documents that Claude Sonnet 5 uses a new tokenizer and
that the same input text yields about 30% more tokens than Claude Sonnet 4.6, while Bedrock
emphasizes that counting is model-specific. Reusing token budgets across model versions is therefore
not safe. 17
A fifth ambiguity is non-canonical tokenization. Recent work shows that the same string can admit
multiple valid tokenizations using a tokenizer’s vocabulary, and that instruction-tuned models can
remain surprisingly robust to these non-canonical encodings. Separate 2025 work shows that such
multiplicity can even create arbitrary price variation in LLM-as-a-service settings, especially for non-
English outputs. For detection engineers, this means “same string” does not always imply “same
billing trajectory” if the generation path changes. 18
A sixth ambiguity is hidden or non-user-visible tokens. OpenAI’s guide notes that local tokenizers miss
model-specific behaviors such as reasoning and caching; Google documents that multimodal preflight
counts can differ from final usage metadata; and recent auditing work shows that hidden reasoning
tokens can dominate cost in opaque reasoning APIs. Even a perfect local text tokenizer may therefore be
unable to explain the final bill. 19
Detection Methodologies
A reliable counting system should not depend on one detector. It should behave like a measurement
instrument with primary, secondary, and fallback channels whose disagreements are themselves
informative.
Static analysis of tokenizer artifacts and code
Static analysis is the strongest route when tokenizer artifacts are available. The goal is to reconstruct the
exact path from input bytes to token IDs by inspecting:
• normalization rules,
• pre-tokenizer configuration,
• tokenizer model type and vocabulary,
• post-processor templates for special tokens,
• model-version mapping.
4

Hugging Face exposes the tokenizer pipeline as explicit components; SentencePiece embeds
normalization choices directly into its model file; and OpenAI’s tiktoken exposes encoding
constructors and special-token maps programmatically. Static inspection can therefore recover not only
count behavior, but also many hidden assumptions about whitespace handling, digit segmentation, and
special-token insertion. Research on BPE merge lists goes further: Hayase et al. show that the ordered
merge list itself leaks information about training-data mixture, demonstrating how much structure a
released tokenizer can reveal. 20
Static analysis is exact for plain-text counting when the released artifacts truly match deployed
inference. Its main weakness is deployment drift: provider-side wrappers, hidden request packing, or
undeclared tokenizer upgrades can invalidate a locally “correct” count. Anthropic’s documented
tokenizer change is a concrete reminder that static assets must be version-pinned and revalidated. 21
Dynamic probing through APIs and model responses
Dynamic probing treats the provider as a black box and infers count behavior from observed outputs
such as token-count endpoints, token listings, or returned usage metadata.
Official provider support for this has improved substantially. OpenAI offers an input-token counting
endpoint for the exact Responses payload and explicitly says it includes roles, boundaries, tools, images,
files, and conversations. Anthropic exposes token counting before message submission. Bedrock
provides a CountTokens operation whose result matches billable usage for the same request. Google
exposes both countTokens and, on Gemini Enterprise Agent Platform, token listing with token IDs plus
a local tokenizer for some models. 22
Dynamic probing is especially valuable for detecting hidden formatting overhead. By submitting
controlled request variants that differ in exactly one factor—one extra role, one extra tool property, one
PDF attachment, one system instruction—you can estimate which structural elements contribute tokens.
OpenAI’s guide warns that local text tokenization misses these structural tokens; Google’s enterprise
tooling goes further by exposing the actual token list for text prompts, making boundary analysis
possible without reverse engineering. 23
Dynamic probing is also how one should detect provider-specific exceptions. Google states that
multimodal counts can be only estimates before execution, and the accurate billable figure arrives in
usage_metadata afterward. Therefore, for multimodal payloads, the dynamic detector should
distinguish preflight estimate, post-execution billed count, and local surrogate estimate as separate
quantities. 24
Hybrid cross-checks, heuristics, and adversarial probes
The most robust architecture is hybrid. Local/static counters provide low-latency deterministic estimates;
provider counters provide the closest thing to ground truth; dynamic probing reveals wrapper overheads;
calibrated heuristics provide fallback coverage when exact methods are unavailable.
Heuristics still matter, but only after calibration. OpenAI and Google both document rough English-
language rules of thumb near four characters per token, yet those are expressly approximate and
language-dependent. They fail badly for CJK text, emoji, mixed code-and-natural-language prompts, or
aggressively normalized Unicode. A calibrated density model should therefore be stratified by script,
format, and domain rather than using one global ratio. 25

Adversarial probes are now a necessity, not an academic luxury. “Broken Tokens?” shows that models
can remain performant under non-canonical tokenizations; “Adversarial Tokenization” shows that
changing token-space representations without changing the visible text can circumvent safeguards; and
TokenBreak demonstrates how token-manipulation attacks can defeat classifier-based guardrails while
leaving downstream semantics intact. A counting detector that passes only easy English prose but fails
on adversarial Unicode, numerical regroupings, or boundary manipulations is not production-ready. 26
Long-text streaming and chunking strategies
Long inputs introduce a different class of error: boundary error. If text is chunked before counting,
merges or whole-token matches that would have crossed a chunk boundary become impossible, so the
sum of per-chunk counts can differ from the true whole-document count. This matters for BPE,
WordPiece, and Unigram alike, though the boundary mechanisms differ. Long-context benchmarks such
as RULER, ∞Bench, LongBench v2, and Needle-in-a-Haystack provide appropriately demanding lengths
and mixed-task structures for testing such effects. 4
When local exact tokenization exists, the right strategy is streaming tokenization or chunking with
seam correction. When exact tokenization does not exist, the right strategy is chunk-level exact API
counting plus sampled seam audits and a global confidence interval. This report formalizes those
methods in the next section.

This framework is preferable to naive averaging because token-count failures are usually structured
rather than random. A local tokenizer may be perfect on plain text yet wrong on tool schemas; a provider
preflight API may be exact on text but only estimated on video; a heuristic may be decent on English

uniformities, so an ensemble that ignores them will miscalibrate uncertainty. 36
A practical discrepancy taxonomy is also useful. Large disagreement between local exact text count and
provider exact payload count usually indicates wrapper overhead. Large disagreement between
provider preflight and post-execution usage suggests multimodal estimation mismatch or caching
effects. Large disagreement between multiple local tokenizers but low provider residuals indicates
family mismatch resolved by calibration. Persistent unexplained residuals after version updates are
drift alarms and should be treated as possible tokenizer changes, as Anthropic’s migration notice
illustrates. 37

Metrics
For count accuracy, measure mean absolute error, median absolute error, signed bias, relative absolute
error, worst-case error, and the 95th and 99th percentiles of absolute error. Signed bias is important
because undercounting and overcounting create different operational risks: truncation failures versus
over-budget rejections. Provider-guided exact counters should be evaluated against output usage
metadata where applicable, especially on Google’s multimodal paths where preflight estimates may
differ from final usage. 24
12

--- PAGE 13 ---
For behavior detection, use precision, recall, and F1 for each hidden behavior class: normalization
present, digit splitting present, wrapper overhead present, tokenizer-family correctly identified, version
drift detected, and multimodal preflight mismatch flagged. For discrepancy scores or anomaly scores,
also report AUROC and PR-AUC. Recent auditing literature on opaque services argues strongly for
separating quantity-verification from semantic-validity verification; the same principle applies to
tokenization detectors. 38
For uncertainty, report nominal versus empirical interval coverage. If a method claims a 95% interval, it
should contain the true count about 95% of the time on held-out documents stratified by language and
format. Also record interval width as a function of length, script mix, and detector availability. This is
where many approximate counters fail in practice: their point estimates can look decent on average while
their uncertainty is badly under-calibrated. 45
For systems performance, measure throughput in bytes/sec or tokens/sec, peak resident memory, API
calls per document, and privacy exposure. This matters because counting is often placed inside routing
or chunking loops. OpenAI’s tiktoken reports 3–6× speedups over a comparable open-source
tokenizer in its README, and Google explicitly notes that local counting can simplify large-prompt
operation because API-based counting may be memory-intensive. 28
Recommended experiments
A rigorous experiment plan should include the following.
First, run a family-identification benchmark. Use short probe strings spanning ASCII, CJK, emoji,
numbers, code, JSON, whitespace variants, and combined marks. Compare candidate local tokenizers
against provider counts using equivalence testing and cross-validated RMSE. This detects whether the
target acts like byte-level BPE, SentencePiece BPE, Unigram, or WordPiece with hidden wrappers. 46
Second, run a normalization and wrapper-ablation benchmark. Use paired strings that differ only in
NFC/NFKC form, full-width forms, repeated spaces, tabs, non-breaking spaces, and leading/trailing
whitespace. Then vary payload shape—single message, multi-message, tools, files, system instructions.
Assess paired significance and structural regression coefficients. 47
Third, run a long-text seam benchmark. On RULER, ∞Bench, and LongBench v2 contexts, compare
whole-document truth against chunked estimates for chunk sizes such as 2K, 8K, and 32K tokens and
overlap sizes such as 32, 64, 128, and 256 bytes or characters. Report seam-bias distributions, not just
averages. 48
Fourth, run an adversarial tokenization suite. Include non-canonical tokenization probes, right-aligned
digit regroupings, character-split strings, prefix-letter perturbations, and safety-wrapper attack strings.
This is necessary both because recent research shows non-canonical tokenizations can preserve
semantics and because safety pipelines can be brittle to token manipulation. 31
Fifth, run a drift-monitoring experiment. Recompute counts weekly or on each model-version change
for a fixed canary corpus. Any systematic shift in affine calibration parameters, residual variance, or
family-classification confidence should raise an alert. Anthropic’s published tokenizer change is exactly
the kind of event this monitor is meant to catch. 21

Performance, memory, and streaming
For text-only paths, use a native or Rust-backed tokenizer whenever possible. OpenAI’s tiktoken
reports a 3–6× speed advantage over a comparable open-source tokenizer in its benchmark setup, which
matters for online chunking and preflight validation loops. For Gemini, Google explicitly documents a
local tokenizer path and notes that remote counting of large prompts can be memory-intensive, so local
counting is preferable whenever it is officially supported. 28
Memory handling should be streaming-first. Large documents should be read from a stream, memory-
mapped if available, and stratified incrementally rather than fully loading and normalizing the whole
document before counting. The implementation should maintain small rolling suffix buffers for seam
correction and avoid duplicate copies of normalized text. SentencePiece-style normalization is
particularly important here because normalization may change whitespace and prefix behavior before
segmentation. 14
Language support
Do not present one global character-to-token ratio as “multilingual support.” Proper language support
means at least script-aware calibration buckets: Latin prose, code, CJK, Arabic/Hebrew with bidirectional
markers, emoji-heavy text, numerals-heavy text, and mixed-format markup. SentencePiece was designed
to operate directly on raw Unicode text for language-independent pipelines, but official English rules of
thumb from providers remain just that—English rules of thumb. The plugin should therefore keep per-
stratum residual statistics and show confidence degradation when a document’s stratum mix differs
from calibration data. 50
Security and privacy
Security defaults should favor local counting first. If remote probing is required, redact identifiers where
possible, sample short windows rather than shipping whole documents, hash cached documents with
model-version salt, and log only aggregate statistics unless explicit audit mode is enabled. This is
especially important because recent auditing research highlights asymmetries between user visibility
and provider-side hidden operations; a counting plugin should not create a second hidden-observability
problem of its own. 38
Limitations, Failure Modes, and Mitigation
The first limitation is conceptual: input token counting is easier than total cost auditing. For input
payloads, provider-native counters can be highly authoritative. For hidden reasoning tokens or internal
tool calls, however, exact validation may be impossible without provider cooperation. CoIn and related
auditing proposals are promising because they combine quantity verification with semantic verification,
but these remain research solutions or cooperation-dependent designs, not drop-in universal fixes. Treat
invisible-token bills as partially observable unless the provider exposes verifiable commitments. 38
The second limitation is model/version drift. Even if your static reconstruction was exact yesterday, an
undeclared back-end change can silently invalidate it tomorrow. Anthropic’s published tokenizer

migration is a documented example, but silent changes can happen elsewhere too. The mitigation is
straightforward: pin versions where possible, maintain a canary corpus, and refuse to label counts as
exact if the observed residual distribution shifts materially. 17
The third limitation is multimodal mismatch. Google explicitly says some multimodal preflight counts
are estimates based on media resolution and may not match final consumed tokens; the accurate billing
usage comes after execution. The mitigation is to represent two numbers in the API—preflight estimate
and post-execution billed count—and never reuse one as the other. 51
The fourth limitation is boundary error on long texts. Any approximate or chunk-local method can
accumulate seam bias, especially when normalization, whitespace-prefix policies, or long merges matter.
The mitigation is seam correction with overlap, sampled seam audits, and interval reporting that
explicitly accounts for residual seam uncertainty. Long-context datasets should be part of the regression
suite for this reason. 52
The fifth limitation is adversarial or non-canonical behavior. Recent papers show that models are more
robust to non-canonical tokenizations than earlier assumptions suggested, and adversarial tokenization
can manipulate safety outcomes without visibly changing text. That means a detector may correctly
count the canonical path while missing meaningful alternate encodings or vulnerabilities in upstream
components such as classifiers and prompt guards. The mitigation is a standing adversarial probe suite
and a policy that no tokenizer integration is “done” until it passes normalization, numeral, Unicode,
and token-manipulation tests. 31
The sixth limitation is privacy leakage through counting itself. Because tokenizer artifacts and counts
can reveal meaningful information—Hayase et al. show merge lists leak training-mixture information—
measurement tools should not be treated as harmless metadata utilities. The mitigation is least-privilege
access to tokenizer assets, redacted telemetry, and local-only modes for sensitive corpora. 53
In practical terms, the safest deployment posture is this: call a count exact only when it is produced by a
provider-native exact counter or a version-matched local tokenizer over the full effective payload; call it
calibrated when it comes from a validated ensemble with empirical coverage; call it heuristic otherwise.
That semantic discipline is as important as any algorithm.
